# 🏃 런닝 훈련 코치 앱 - 프로젝트 기획 문서

> 최종 업데이트: 2026-02-28
> 상태: 기획 진행 중

---

## 1. 프로젝트 개요

### 1.1 목적
사용자의 실제 운동 데이터(Apple HealthKit)를 기반으로 개인화된 런닝 훈련표를 생성하고, 훈련 달성도를 추적·판단해주는 모바일 앱

### 1.2 핵심 기능
- HealthKit 연동을 통한 운동 데이터 수집
- 목표 기록 설정 (레이스 목표 / 습관 목표)
- VDOT 기반 개인화 훈련표 생성
- LLM을 활용한 훈련표 개인화 조정 및 자연어 설명
- 훈련 달성도 추적 및 판단

## 2. 기술 스택 결정사항

### 2.1 모바일 프레임워크: Flutter (iOS 우선)

| 항목 | 결정 | 비고 |
|------|------|------|
| 프레임워크 | Flutter | 과거 학습 경험 있음 |
| 타겟 플랫폼 | iOS 우선 | 본인 테스트 기기: iPhone |
| Android 확장 | MVP 이후 | Health Connect 연동 별도 작업 필요 |

**근거:**
- 크로스플랫폼이므로 UI 코드 재사용 가능 (추후 Android 확장 비용 낮음)
- HealthKit은 iOS 전용이므로 양 플랫폼 동시 개발은 스코프 과다
- Android의 Health Connect는 데이터 구조·권한 모델이 다름

### 2.2 HealthKit 데이터 연동 조사 결과

#### Flutter `health` 패키지 (cph-cachet)

**지원 확인된 런닝 관련 데이터:**

| 데이터 | HealthDataType | 지원 여부 |
|--------|---------------|-----------|
| 워크아웃 (런닝) | `WORKOUT` (RUNNING) | ✅ |
| 총 거리 | `WorkoutHealthValue.totalDistance` | ✅ |
| 총 소모 칼로리 | `WorkoutHealthValue.totalEnergyBurned` | ✅ |
| 심박수 | `HEART_RATE` | ✅ |
| 걸음 수 | `STEPS` | ✅ |
| 걷기/달리기 거리 | `DISTANCE_WALKING_RUNNING` | ✅ |
| 걷기 속도 | `WALKING_SPEED` | ✅ (iOS only) |
| 워크아웃 루트 (GPS) | `WORKOUT_ROUTE` | ✅ (쓰기 지원, 읽기는 제한적) |
| 걷기 심박수 평균 | `WALKING_HEART_RATE` | ✅ (iOS only) |

**⚠️ 제한사항 / 미지원 가능성:**

| 데이터 | 상태 | 대안 |
|--------|------|------|
| 러닝 케이던스 (steps/min) | ❓ 별도 `RUNNING_CADENCE` 타입 없음 | HealthKit 자체에 `runningCadence` 타입이 없음. 구간 걸음수/시간으로 계산 가능 |
| 구간별 페이스 (splits) | ❌ 직접 제공 안 함 | 워크아웃 루트 GPS + 타임스탬프로 직접 계산 필요 |
| 러닝 스피드 | ✅ `WALKING_SPEED` (러닝에도 적용) | HealthKit의 `.walkingSpeed`가 러닝 포함 |
| 러닝 파워 | ❓ 패키지에서 미확인 | HealthKit에는 `runningPower` 존재하나 Flutter 패키지 지원 미확인 |
| 러닝 보폭 | ❓ 패키지에서 미확인 | HealthKit에는 `runningStrideLength` 존재하나 Flutter 패키지 지원 미확인 |
| 지면 접촉 시간 | ❓ 패키지에서 미확인 | HealthKit에는 `runningGroundContactTime` 존재 |
| 수직 진동 | ❓ 패키지에서 미확인 | HealthKit에는 `runningVerticalOscillation` 존재 |

#### 결론 및 권장사항

**Flutter `health` 패키지로 MVP 진행 확정.**

핵심 데이터(워크아웃, 거리, 심박수, 속도)는 지원됨. 러닝 케이던스, 구간별 페이스 등 세부 메트릭은 있으면 좋지만 MVP 필수는 아님.

**세부 데이터 부족 시 대응:**
- Flutter Method Channel로 네이티브 HealthKit API 직접 브릿지 (Flutter 유지하면서 Swift 코드로 필요한 데이터만 접근)
- React Native / Swift 네이티브 전환은 하지 않음 ❌ (확정)

### 2.2.1 Strava API 연동 (선택적)

**Strava API는 무료.** 개발자 등록 무료, API 호출 무료. rate limit(15분당 200요청, 일 2,000요청)만 존재.

#### 데이터 수집 전략

```
사용자 데이터 수집
  ├→ 기본: HealthKit (모든 iOS 사용자, 연동 필수)
  └→ 선택: Strava API (연동 활성화 시, 더 풍부한 데이터)
```

| 항목 | HealthKit | Strava API |
|------|-----------|------------|
| 대상 사용자 | 모든 iOS 사용자 | Strava 계정 보유자 |
| GPS 루트 | 제한적 | 상세 제공 (polyline + 고도) |
| 구간별 페이스 | 직접 계산 필요 | 랩/스플릿 데이터 제공 |
| 고도 데이터 | 패키지 지원 미확인 | 제공 |
| 심박수 | 시계열 데이터 제공 ✅ | 평균/최대만 제공 |
| 케이던스 | 간접 계산 필요 | 평균 케이던스 제공 |
| 인증 | HealthKit 권한 요청 | OAuth2 |

#### 데이터 우선순위 로직

```
if (Strava 연동 활성화) {
  운동 기록 → Strava API에서 가져옴 (구간 페이스, 고도, 케이던스 포함)
  심박수 시계열 → HealthKit에서 보완 (Strava는 상세 심박수 미제공)
} else {
  운동 기록 → HealthKit에서 가져옴 (기본 데이터)
}
```

#### Strava 연동의 추가 가치
- **대회 준비 모드:** 사용자가 대회 코스를 한 번이라도 달린 Strava 기록이 있으면 고저도 프로필 자동 확보 가능
- **더 정확한 훈련 분석:** 구간별 페이스, 고도 기반 노력도 분석
- **데이터 이중 백업:** HealthKit + Strava 두 소스에서 데이터 확보

> **참고:** 나이키 런클럽은 공개 API를 제공하지 않음 (2024년 3월 비공식 API도 차단됨). NRC 사용자는 NRC → Apple Health 또는 NRC → Strava 자동 동기화를 통해 간접적으로 데이터 접근.

### 2.3 인증: Supabase Auth + OAuth (소셜 로그인 전용)

| 항목 | 선택 | 비고 |
|------|------|------|
| Apple Sign-in | 🔴 필수 | iOS 앱 소셜 로그인 제공 시 App Store 심사 정책상 필수 |
| Google Sign-in | 🔴 필수 | 한국 사용자 대상 주요 소셜 로그인 |
| Kakao 로그인 | 🔴 필수 | 한국 서비스 대상 핵심 소셜 로그인 |
| Email/Password 회원가입 | ❌ 없음 | 소셜 로그인만 지원 |

**참고사항:**
- Apple, Google은 Supabase Auth에서 기본 지원
- Kakao는 Supabase Auth에서 기본 제공하지 않음 → Custom OAuth Provider로 직접 연동 필요 (추가 공수 있음)
- 별도 회원가입 기능 없이 소셜 로그인만으로 계정 생성/로그인 처리

## 3. 훈련표 생성 로직

### 3.1 VDOT 시스템이란?

운동생리학자 Jack Daniels 박사가 개발한 런닝 능력 평가 시스템.

**핵심 개념:**
- 사용자의 최근 레이스 기록(5K, 10K, 하프, 풀마라톤)을 입력하면 **VDOT 점수**를 산출
- VDOT 점수를 기반으로 **5가지 훈련 페이스 존**을 자동 계산:

| 페이스 | 이름 | 목적 | 설명 |
|--------|------|------|------|
| E | Easy | 기초 지구력 | 조깅 페이스. 전체 훈련의 대부분을 차지 |
| M | Marathon | 레이스 적응 | 마라톤 레이스 페이스. 심리적 훈련 목적 |
| T | Threshold | 젖산 역치 향상 | 템포런. 약 20~30분 유지 가능한 강도 |
| I | Interval | VO2max 향상 | 인터벌. 약 3~5분 유지 가능한 강도 |
| R | Repetition | 스피드·효율 | 반복달리기. 짧고 빠른 구간 |

**예시:** VDOT 40인 러너의 경우
- E 페이스: 5'54"~6'29"/km
- M 페이스: 5'15"/km
- T 페이스: 4'55"/km
- I 페이스: 4'32"/km
- R 페이스: 4'17"/km

**VDOT의 역할 (이 앱에서):**
- 과거 훈련 데이터가 없는 사용자의 현재 수준 추정에 활용
- LLM에게 전달하는 기초 데이터(페이스 존)의 신뢰성 확보
- 입력이 단순 (레이스 기록 하나면 충분), 출력이 명확 (5가지 페이스 자동 산출)
- 과학적으로 검증됨 (50년 이상 연구 기반)

### 3.2 사용자 데이터 입력 방식

#### 입력 경로 A: 과거 운동 데이터 있음 (HealthKit 연동)
```
HealthKit 운동 데이터 수집
  → 최근 N주 러닝 요약 생성 (주간 볼륨, 평균 페이스, 심박수 패턴)
  → 사용자가 목표 기록 입력 (예: 하프마라톤 1:55)
  → 사용자가 주간 가용 훈련일수 입력
  → [선택] 대회 기록증 이미지 업로드 또는 대회 기록 직접 입력
  → VDOT 점수 계산 (룰 기반)
  → 모든 데이터를 LLM에 context로 전달
  → LLM이 개인화된 훈련표 생성
```

#### 입력 경로 B: 과거 운동 데이터 없음
```
사용자가 목표 기록 입력
  → [선택] 대회 기록증 이미지 업로드 또는 대회 기록 직접 입력
  → [선택] 주간 가용 훈련일수, 현재 러닝 빈도/거리 자가 입력
  → VDOT 점수 계산 (대회 기록 기반) 또는 초보자 기본값 적용
  → 데이터를 LLM에 context로 전달
  → LLM이 훈련표 생성 (데이터가 적은 만큼 보수적으로)
```

#### 대회 기록 입력

| 항목 | 내용 |
|------|------|
| 기본 방식 | 사용자가 직접 입력: 날짜 + 대회 이름 + 거리 + 완주 시간 |
| 저장 | Supabase에 대회 기록 테이블로 관리 |
| 활용 | VDOT 점수 산출, 성장 추이 분석 |

> **대회 기록증 이미지 OCR 인식은 추후 논의.** MVP에서는 수동 입력만 지원.

### 3.3 훈련표 생성 아키텍처: 하이브리드 (룰 기반 계산 + LLM 생성)

**결정: VDOT 계산은 룰 기반, 훈련표 구성·코칭은 LLM**

```
┌─────────────────────────────────────────────────┐
│                  사용자 입력                       │
│  목표기록, 대회기록, 훈련일수, HealthKit 데이터      │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│            룰 기반 계산 레이어 (코드)               │
│  - VDOT 점수 산출                                 │
│  - 5가지 훈련 페이스 존 계산                        │
│  - HealthKit 데이터 요약 (주간 볼륨, 빈도, 페이스)    │
│  - 사용자 수준 분류 (초보/중급/고급)                  │
└──────────────────────┬──────────────────────────┘
                       │ structured context
                       ▼
┌─────────────────────────────────────────────────┐
│              LLM 생성 레이어 (API 호출)             │
│  - 주차별·요일별 훈련표 구성                         │
│  - 훈련 볼륨·강도 배분 (사용자 수준 반영)              │
│  - 각 훈련의 목적·방법 자연어 설명                    │
│  - 주간 코칭 메시지 생성                             │
│  - 달성/미달성에 대한 피드백 생성                      │
└──────────────────────┬──────────────────────────┘
                       │
                       ▼
┌─────────────────────────────────────────────────┐
│              출력: 개인화 훈련표 + 코칭               │
└─────────────────────────────────────────────────┘
```

**이 방식을 채택한 이유:**

| 역할 | 담당 | 이유 |
|------|------|------|
| VDOT 계산, 페이스 존 산출 | 룰 기반 (코드) | 수학 공식이므로 정확도가 중요. LLM에게 맡기면 오히려 부정확 |
| HealthKit 데이터 전처리 | 룰 기반 (코드) | 데이터 파싱·집계는 코드가 적합 |
| 훈련표 구성 (어떤 날에 어떤 훈련을) | LLM | 사용자 상황(볼륨, 빈도, 목표)에 따른 유연한 배분 |
| 코칭 메시지 | LLM | 자연어 생성, 동기부여, 개인화된 피드백 |
| 달성도 판단 피드백 | LLM | "이번 주 장거리 런이 부족합니다" 같은 맥락적 분석 |

### 3.4 LLM 코칭 메시지 기능

| 기능 | 설명 | 토큰 비용 고려 |
|------|------|----------------|
| 훈련표 생성 시 설명 | 각 훈련의 목적과 방법을 자연어로 설명 | 훈련표 생성 시 1회 호출 |
| 주간 리뷰 | 이번 주 달성도 분석 + 다음 주 조언 | 주 1회 호출 |
| 달성/미달성 피드백 | 개별 훈련 완료 후 피드백 | 빈도 높음 → 간결하게 제한 또는 배치 처리 |
| 훈련표 재조정 제안 | 컨디션·달성도에 따라 훈련표 수정 제안 | 필요 시에만 호출 |

**토큰 비용 관리 전략:**
- 실시간 채팅 코칭은 MVP에서 제외 (비용 대비 효과 불확실)
- 코칭 메시지는 **비동기 생성** (사용자가 요청할 때, 또는 주간 리뷰 시점에 생성)
- 개별 훈련 피드백은 짧은 프롬프트 + 짧은 응답으로 제한
- 추후 사용 패턴을 보고 대화형 코칭 도입 여부 판단

### 3.5 LLM 활용 방식

**결정: RAG 미사용 확정, 직접 API 호출**

| 방식 | 채택 여부 | 이유 |
|------|-----------|------|
| RAG (논문/문서 기반) | ❌ 미사용 | 문서 수집·임베딩·관리 부담, 1인 개발에서 투입 대비 효과 낮음 |
| LangChain/LangGraph | ❌ MVP에서 불필요 | 현재 워크플로우는 단일 프롬프트 → 응답 구조로 충분 |
| **직접 API 호출 (채택)** | ✅ | 구현 간단, 프로바이더 변경 용이, 충분한 유연성 |

**LLM 프로바이더 추상화 설계:**
```
LLMService (인터페이스)
  ├── OpenAIService (1차)
  ├── ClaudeService (추후 가능)
  └── 기타 (확장 가능)
```
- API 키와 모델명을 환경변수로 관리
- 요청/응답 포맷을 통일하여 프로바이더 교체 시 코드 변경 최소화

**LangChain/LangGraph 재검토 시점:**
- 사용자와 멀티턴 대화형 코칭이 필요해질 때
- 여러 데이터 소스(HealthKit + 날씨 + 코스 정보 등)를 조합한 복합 분석이 필요할 때
- 현재는 해당 복잡도가 없으므로 직접 호출로 충분

### 3.6 날씨 연동 (차별화 기능)

**배경:** 기온은 러닝 퍼포먼스에 직접적인 영향을 미침. 15°C 전후가 최적이며, 고온·저온에서는 동일한 노력으로도 페이스가 느려지는 게 정상. 기존 런닝 앱들은 이를 고려하지 않아 사용자가 "왜 오늘은 느리지?"라는 불필요한 좌절을 경험함.

**활용 방안:**

| 기능 | 설명 | 예시 |
|------|------|------|
| 훈련 페이스 보정 | 당일 기온·습도에 따라 목표 페이스 자동 조정 | "오늘 32°C, E 페이스 6'00" → 6'15"~6'30"으로 조정" |
| 훈련 일정 제안 | 주간 날씨 예보 기반 고강도 훈련 배치 최적화 | "수요일이 선선하니 인터벌을 수요일로 이동 추천" |
| 기온 보정 기록 분석 | 과거 기록을 기온 보정하여 실제 성장 추이 파악 | "7월 페이스 하락은 기온 영향, 실력은 유지 중" |
| 날씨 경고 | 극한 기온·미세먼지 시 실내 훈련 권고 | "내일 체감온도 38°C, 실내 러닝이나 크로스트레이닝 권장" |

**기술 구현:**

| 항목 | 선택지 |
|------|--------|
| 날씨 API | OpenWeatherMap (무료 티어 충분) 또는 기상청 API |
| 데이터 | 기온, 체감온도, 습도, 풍속, 미세먼지 |
| 적용 방식 | 날씨 데이터를 LLM context에 포함하여 훈련표 생성 시 반영 |

**MVP vs 추후:**
- **MVP:** 당일 기온 기반 페이스 보정 메시지 (날씨 API → LLM context 추가, 현재 아키텍처로 충분)
- **추후:** 주간 예보 기반 훈련 일정 최적화, 기온 보정 기록 분석 (복잡도 증가 시 LangGraph 검토)

> 💡 **차별화 포인트:** 대부분의 런닝 앱이 날씨를 고려하지 않음. 이 기능만으로도 여름/겨울 훈련에 대한 실질적인 가치를 제공할 수 있음.

## 4. 사용자 데이터 기반 개인화 범위

### 4.1 MVP에서 활용할 데이터

| 데이터 | 활용 방법 | 우선순위 |
|--------|-----------|----------|
| 최근 레이스 기록 | VDOT 점수 산출 → 훈련 페이스 결정 | 🔴 필수 |
| 총 러닝 거리 | 주간 볼륨 추정 → 훈련량 설정 | 🔴 필수 |
| 러닝 빈도 | 주간 훈련일수 추정 | 🟡 권장 |
| 평균 심박수 | 훈련 강도 검증 (페이스 존 크로스체크) | 🟡 권장 |
| 러닝 속도/페이스 | 현재 수준 파악, VDOT 보정 | 🟡 권장 |

### 4.2 데이터 활용 단계

#### MVP: 데이터 수집 + 훈련표 생성 context로 활용

Strava 연동 활성화 시 아래 데이터를 **수집·저장하고 LLM context로 전달:**

| 데이터 | 소스 | 활용 |
|--------|------|------|
| 케이던스 | Strava API | 훈련표 생성 시 참고 데이터로 LLM에 전달 |
| 구간별 페이스 (splits) | Strava API | 페이스 균일도를 LLM이 판단하여 코칭 반영 |
| 고도 데이터 | Strava API | 업힐/다운힐 구간 고려한 훈련 조언 |
| 심박수 시계열 | HealthKit | 심박수 존 분석, 훈련 강도 검증 |

#### 고도화: 별도 분석 UI 구축

| 기능 | 설명 |
|------|------|
| 케이던스 추이 차트 | 주/월별 케이던스 변화 시각화 |
| 구간 페이스 균일도 분석 | 레이스에서 이븐/네거티브 스플릿 판단 |
| 심박수 존 분포 | 훈련별 심박수 존 시간 비율 시각화 |
| 고도 기반 노력도 분석 | 평지 환산 페이스 계산 |

### 4.3 데이터가 없는 사용자

- 목표만 입력 (예: "하프마라톤 완주")
- 일반적인 초보자 훈련표 템플릿 제공
- 훈련 데이터가 쌓이면 점진적으로 개인화

## 8. 추후 로드맵 (MVP 이후)

### 8.1 대회 준비 모드 / 대회 정보 허브

**배경:** 대회별 고저도, 과거 날씨, 코스 특성 등의 정보가 흩어져 있어 러너가 대회를 준비할 때 불편함. 이를 한곳에 모아 대회 맞춤 전략을 제공하는 기능.

**핵심 기능:**

| 기능 | 설명 |
|------|------|
| 대회 코스 고저도 프로필 | 대회별 코스 고저도 시각화 (업힐/다운힐 구간 표시) |
| 고저도 기반 페이스 배분 전략 | "1~5km 오르막 구간은 M+15초, 15~20km 내리막은 M-10초" 같은 구간별 페이스 가이드 |
| 대회 과거 날씨 | 최근 3~5년 대회 당일 기온·습도·풍속 기록 |
| 올해 예상 날씨 비교 | 과거 데이터 + 올해 예보 비교 → 페이스 보정 제안 |
| 대회 종합 전략 | LLM이 고저도 + 날씨 + 사용자 VDOT을 종합하여 레이스 전략 생성 |

**데이터 확보 방안:**

| 데이터 | 소스 | 비고 |
|--------|------|------|
| 코스 고저도 | 대회 공식 홈페이지 코스맵/고저도 이미지, GPX 파일 확보 시 Google Elevation API로 정밀 고도 보완 | 한국 주요 대회(서울, JTBC, 춘천, 경주, 동아 등) 10~20개부터 수동 구축 |
| 과거 날씨 | 기상청 과거 관측 API (또는 기타 날씨 API) | API 제공되므로 기술적 난이도 낮음 |
| 대회 기본 정보 | 수동 큐레이션 (날짜, 장소, 거리, 코스 특성) | 매년 업데이트 필요 |

**스코프:** 한국 주요 대회 위주로 시작 (확정)

**기술 참고:**
- 이 기능이 추가되면 "날씨 API + 고저도 데이터 + 사용자 VDOT + HealthKit 데이터"를 조합하는 복합 분석이 필요 → LangGraph 도입 검토 시점
- Google Elevation API로 위경도 기반 고도 데이터 조회 가능 (GPX 파일의 좌표를 활용)
